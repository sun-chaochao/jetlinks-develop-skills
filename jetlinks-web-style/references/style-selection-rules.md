# 页面交互方案选择规则

## 默认行为：强制建立方案档案

`jetlinks-web` 前端任务的默认行为是：**先建立交互方案档案，再进入实现**。任何"新增页面 / 重构页面壳层 / 调整信息架构 / 调整主筛选或主列表 / 改变首屏第一任务"都必须命中本规则。用户确认不是每次都必须等待：当业务事实清楚且只有一个可信推荐方案时，可以记录为"默认采用推荐方案"并继续实现；当关键事实缺失、结构风险高，或 2 个以上方案同样合理时，必须先让用户选择。

交互方案的**索引与分片加载**见 [`style-catalog.md`](style-catalog.md)：选型时优先读 `style-catalog-routing.md` + **`style-catalog-core-base.md`**；若涉及对象详情 / 主从 / 配置态-运行态 / 侧栏壳，再读 **`style-catalog-core-detail-shell.md`**；**不要**把 `style-catalog-templates.md`（13 个方案全文）整篇加载进上下文，只读与用户候选对应的 `###` 小节。

除非命中下面的"局部调整白名单"，否则不允许跳过方案档案直接进入编码。

## 局部调整白名单

只有任务被明确判定为下列局部调整时，才允许跳过方案选择直接实现：

1. 仅修改 / 新增 / 删除一个表单字段
2. 仅修改 / 新增 / 删除一个筛选条件
3. 仅修改一个弹窗、抽屉或单个区块的内容与文案
4. 仅样式、文案、组件 `props` 微调（含 i18n key 替换、按钮文案、占位文案、图标替换）

边界：

- 改一个字段但同时影响列表列、详情分区与编辑弹窗 → 不再是局部调整，必须先建立方案档案
- 改一个弹窗但同时调整页面壳层、Tab 结构或首屏组织 → 不再是局部调整
- 改一个筛选条件但需要新增 `QuickFilterSidebar` 或调整主筛选模型 → 不再是局部调整

## 触发选择的硬场景

下列场景命中任一项时，必须走完整方案档案流程：

- 新增页面
- 页面壳层 / 信息架构 / 主筛选 / 主列表 / 主详情承载方式调整
- 用户明确说"参考现在这个页面的风格"、"沿用资产页这种结构"、"按当前模板"
- 用户要求"更现代"、"更适合运营使用"、"不要普通表格页"、"换种交互方式"
- 列表、筛选、详情、编辑方式、资源选择方式存在多个合理做法且会影响用户路径
- 页面在"标准表格 / 工作台 / 台账 / 主从 / 详情工作区 / 配置向导 / 监控感知 / 钻取 / 时间线 / 对比"等方案之间存在分歧

## 选择流程

### 1. 先做业务任务归类

按 [`style-catalog-routing.md`](style-catalog-routing.md) 中的「业务任务 → 方案路由」表先确定第一任务：

- 浏览定位一批对象
- 横向对比多个对象
- 处置状态事件
- 持续维护单个对象
- 配置开通新能力
- 选择资源 / 模型 / 模板
- 监控整体态势
- 分析指标与维度
- 查看事件演进 / 操作历史

任务归类不清楚时，先问用户"进入页面后第一步要做什么"，不要直接默认。

### 2. 只给少量具体业务选项

给 1 个推荐方案和最多 3 个替代方案，每个选项必须按 [`style-catalog-routing.md`](style-catalog-routing.md) 中「方案统一字段」说明：

- 业务任务匹配理由
- 首屏结构（ASCII 线框骨架要点）
- 信息密度目标
- 编辑触发梯度
- 核心组件（来自 `jetlinks-web-core`）
- 状态 / 标签锚点
- 主要动作位置
- 不借鉴什么

不要问：

- "你想要什么风格？"
- "你喜欢哪种设计？"
- "你能再描述一下吗？"

应改成：

- "进入这个页面后用户的第一任务是 X；我建议按 A 继续，B / C 是可选替代。若你没有异议，我会采用 A。"

### 3. 确认模式必须明确

如果用户没有额外偏好，给出一个默认项并说明业务理由。按以下规则决定是否等待用户：

- 已知事实能明确支持一个方案，且用户要求实现 → 记录"默认采用推荐方案"并继续实现。
- 关键事实缺失、页面骨架影响大、或 2 个以上方案同样合理 → 等用户选择后再实现。

推荐表达：

- "如果你没有特别偏好，我默认按 `资产卡片台账页` 继续，因为这个页面的第一任务是快速识别资产状态并进入详情处置。"
- "这里 `资产卡片台账页` 和 `筛选工作台页` 都合理，但会改变首屏主任务；请先确认一种。"

### 4. 固化 solution profile

一旦方案被用户选定或被记录为默认采用，后续实现都围绕这个 profile，不要在实现途中再次漂移。

最少记录这些项：

- solutionName
- primaryFilterSurface
- contentSurface
- detailCarrier
- editMode
- coreComponents
- actionPlacement
- densityTarget
- sidebarMode
- confirmationMode
- rejectedAlternatives

### 5. 方案选择不替代组件核验

- 方案选择只锁定业务结构和交互路径。
- 具体组件仍必须回到目标 workspace 的 `jetlinks-web-core` 核验真实导出、props、slots、emits 和相邻用法。
- 不要因为某个参考页面手写了局部组件，就在当前业务里复制一份。
- 如果用户明确要求参考 `jetlinks-project-ui-cli`，只能把它作为外部参考，不作为默认依赖或导入来源。

### 6. 方案档案建立后必须遵守反传统后台感硬约束

细则以 [`style-catalog-core-base.md`](style-catalog-core-base.md) 全文为准；涉及对象详情 / 主从 / 配置态-运行态 / 侧栏时再读 [`style-catalog-core-detail-shell.md`](style-catalog-core-detail-shell.md)。至少核对：

- 编辑触发梯度 `inline > sectional > drawer > modal > page` 必须按业务最小代价优先评估
- 首屏不允许默认叠加"4 KPI 卡 + 顶部搜索 + 大表格"；一屏一主视觉锚点
- 列表 / 卡片第一信息层必须是状态、标签、归属、主动作；状态强调收口（禁单侧彩色线条为主要信号）
- 信息密度按 catalog 给出的目标执行
- 文案面向终端用户，不残留开发态文字
- 弹窗不是唯一编辑路径；单对象详情须遵守 detail-shell 中 §7 摘要区与 §8 编辑样式统一

**入口表**见 [`style-catalog-core.md`](style-catalog-core.md)。

## 现有映射规则

### 用户说"按现在资产页这种风格"

直接映射为：

- `资产卡片台账页` 或 `筛选工作台页`

根据第一任务决定：

- 快速识别对象状态并进入详情：`资产卡片台账页`
- 高频组合条件定位记录：`筛选工作台页`

### 用户说"就普通后台表格那种"

直接映射为：

- `标准管理表格页`

同时提醒：是否真的需要表格？如果对象有强状态识别 / 持续筛选 / 处置流，建议改走 `资产卡片台账页` / `筛选工作台页` / `处置工作台`。如果最终仍选表格页，搜索层默认仍是 `ConditionFilter`；只有窄改旧页、用户明确要求旧表格风格，或筛选只是少量固定字段且无路由回显 / 远程选项 / 保存搜索需求时，才使用 `ProSearch` 并写明例外理由。

### 用户说"左侧树 / 列表，右侧详情编辑"

直接映射为：

- `主从详情工作区`

### 用户说"告警 / 工单 / 巡检处理"或"处置类页面"

直接映射为：

- `处置工作台`

### 用户说"几个对象放一起看 / 对比"

直接映射为：

- `多对象对比页`

### 用户说"看整体态势 / 健康度 / 分布"

直接映射为：

- `监控感知页` 或 `分析钻取页`（看是否需要按维度下钻）

### 用户说"按时间看事件 / 日志 / 回放"

直接映射为：

- `时间线 / 日志流页`

### 用户说"看运行状态又能改配置"

直接映射为：

- `配置态-运行态切换页`

### 用户说"选模型 / 选模板 / 选能力"

直接映射为：

- `资源 / 能力选择器`

## 扩展规则

后续新增方案时，不改主流程，只在 [`style-catalog-templates.md`](style-catalog-templates.md) 中增加 `###` 小节：

- 方案名
- 业务任务
- 适用场景 / 不适用
- 首屏 ASCII 线框骨架
- 信息密度目标
- 编辑触发梯度
- 核心组件
- 状态 / 标签锚点
- 主要动作位置
- 不借鉴清单
- 与相近方案的分界

# JetLinks MBean 运维可观测性规则

本文件用于常驻内存任务、缓存、队列、缓冲区、会话管理器、订阅管理器、重试池、连接池、协议支持管理等长期运行能力的运维可观测性设计。目标是让运维人员能快速看到状态、定位积压或错误，并在安全范围内执行刷新、重试、清理等内部操作。

## 什么时候需要 MBean

新增或修改以下能力时，主动判断是否需要 MBean：

- 本地内存状态：缓存、索引、路由表、订阅树、会话表、连接表、任务注册表。
- 后台执行链路：定时任务、批处理、异步消费、重试队列、死信队列、持久化 buffer。
- 资源池或运行时组件：线程池、连接池、Netty allocator、协议支持、网关、外部客户端。
- 运维需要在不改数据、不重启服务的情况下查看状态或触发内部动作。

不需要新增 MBean 的常见情况：纯无状态请求链路、一次性工具逻辑、已有同模块 MBean 已覆盖，或平台已有更合适的监控/命令服务入口。

## 暴露什么

优先暴露能辅助定位问题的低成本信息：

- 统计：总量、当前大小、积压数、死信数、命中/未命中、输入/输出、成功/失败、丢弃数。
- 监控：最近错误、最后更新时间、耗时分布、处理次数、wip、存储大小、配置摘要、节点/订阅摘要。
- 内部操作：`refresh`、`reload`、`resetMonitor`、`flush`、`compact`、`retryDead(maxSize)`、`clearExpired`、`removeLocal`。

不要暴露 token、密钥、完整 payload、完整文件内容、隐私字段、无限集合或未脱敏用户输入。查看类操作必须限制返回数量，例如 `peekDead(size)` 要设置上限。

## 实现约束

- 优先复用相邻模块已有 MBean 模式。常见形式是 `ManagementFactory.getPlatformMBeanServer()`、`ObjectName("org.jetlinks:type=<Type>,name=<Name>")`、`StandardMBean(new XxxMBeanImpl(), XxxMBean.class)`。
- `ObjectName` 使用稳定 domain `org.jetlinks`；`type` 表示能力类型，`name` 表示实例或实现名。动态 name 必须转义或替换空格、斜杠、冒号、通配符等非法字符。
- 生命周期要完整：初始化时注册，关闭时注销；重复注册、注册失败或注销失败只记录 warning，不影响主业务启动和关闭。
- MBean 接口只暴露运维语义，不暴露内部可变集合引用；实现类优先私有化或静态化，避免扩大业务 API 面。
- 操作类方法必须有边界：参数有上限，行为幂等或可重复执行，耗时操作不阻塞响应式事件循环。
- 会改变运行状态的操作要可解释：方法名体现影响范围，必要时记录日志或 TraceHolder span。
- 不把 MBean 当业务接口；如果需要租户权限、审计、审批或用户态操作，应设计受控的命令 / API，而不是直接扩展 MBean。

## 设计稿里写什么

涉及常驻内存、缓存、后台任务或重试能力时，设计稿写明：

- 是否需要 MBean；不需要时说明已有覆盖或不适用原因。
- MBean 名称：`org.jetlinks:type=...,name=...`。
- 统计指标：数量、积压、错误、耗时、资源占用、配置摘要等。
- 运维操作：刷新缓存、手动重试、清理过期、compact、重置统计等。
- 安全边界：敏感信息排除、返回数量上限、操作参数上限、是否可能影响业务。

## 测试目标

- 注册和注销不会影响主业务生命周期。
- 指标能随已知操作变化，并能反映积压、错误或缓存命中等关键状态。
- 运维操作有边界：不会无限返回数据，不会越权暴露敏感内容，不会绕过业务不变量。
- 如果通过平台 MBeanCommand / InvokeMBeanCommand 暴露给运维入口，验证命令能读取属性并调用允许的操作。

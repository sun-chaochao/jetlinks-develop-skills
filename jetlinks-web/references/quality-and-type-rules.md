# JetLinks Web Quality And Type Rules

本文件整合交付质量约束与 TypeScript 约束，作为前端改动的统一兜底规则。

## 适用范围

- 页面或模块改造后的交付前自检
- 需要控制 `any` 扩散和断言风险的改动
- Code Review 前的质量与类型一致性检查

## 必须遵守

- 优先复用 `@jetlinks-web-core` 与 `@jetlinks-web` 的现有能力。
- API 请求统一落 `api/*.ts`，避免页面内硬编码接口。
- 复杂逻辑优先抽离到 hook 或 service。
- 新增能力前先确认是否已有同类实现可复用。
- 关键边界补齐类型：API 入参/出参、组件 props、store state、路由参数。
- 用户可见文案统一走当前工作区 i18n 机制，避免在页面、schema、columns 或状态映射中散落硬编码标题、字段展示名、按钮文本和枚举文案。
- 前端代码不要求通篇注释，但关键业务判断、兼容逻辑、非显而易见的状态联动和公共 API 契约必须补充简洁注释。

## 样式 Token / 变量优先规则

- 样式优先使用相邻页面模式、现有 Design System、Ant Design Vue token、已有 less/css 变量和公共 class。
- 新增颜色、阴影、圆角、间距、渐变前，先确认项目是否已有变量、组件 prop、公共 class 或相邻页面模式可复用。
- 状态色、标签色、告警色、平台差异色优先使用已有枚举映射、模块常量或统一组件能力。
- 避免在业务页面大量散落 hex 色值、魔法间距、一次性阴影、临时渐变和重复动画。
- 需要新增视觉变量时，应优先放到模块统一样式、公共变量或明确的局部组件中，不在多个页面重复定义。
- 样式只服务当前业务差异；如果变成跨页面模式，应收敛为局部组件、模块组件或公共样式能力。

## 视觉反例约束

以下写法默认不推荐，除非相邻页面已有一致模式或需求明确要求：

- 不要在单个业务页面临时创造新的卡片体系、搜索体系、弹窗体系、状态标签体系或大面积视觉风格。
- 不要绕开 Ant Design Vue、`@jetlinks-web-core` 或 `@jetlinks-web` 组件，手写同类基础控件。
- 不要为一个页面单独写一套状态颜色、标签颜色、按钮颜色或图标语义。
- 不要用大量深层选择器强行覆盖组件库样式；优先使用组件 props、class、slot 或相邻页面已有写法。
- 不要在多个组件中复制同一段卡片头部、操作栏、空状态、loading、状态标签或查询条件模板。
- 不要为了“看起来更丰富”添加与项目设计体系不一致的阴影、渐变、边框、背景图或动画。
- 不要在页面内写只能服务一次的复杂 CSS hack；确需处理兼容问题时应收敛到局部组件并说明原因。

## 单一真理源约束

- 同一个业务枚举、状态文案、状态颜色、图标映射、权限标识、路由参数名或接口字段映射，不得在多个组件中重复定义。
- API 入参/出参类型优先收敛到 `api/*.ts`、模块 model、types 文件或已有类型定义中。
- 搜索条件、表格列、卡片字段、详情字段如果多处共享，优先抽为配置项、hook 或模块 constants。
- 权限按钮、菜单、动态路由、token、base API、上传 header 等平台能力必须复用已有机制，不在页面内硬编码。
- 多个组件展示同一个数字、名称、状态或统计口径时，必须读取同一个 store、hook、computed 或统一接口结果。
- 发现显示不一致时，优先修正统一来源，不要分别修改各个组件里的展示逻辑。
- Mock、静态 options 和临时映射如果未来会接接口，应标明边界并集中放置，避免散落在模板和事件函数中。

## Web Code Generation Constraints

以下约束用于补充页面生成、组件拆分与交付质量规则：

1. JSX/Vue 文件接近或超过 300 行时必须做职责复查，但 300 行不是硬性拆分线；只有出现混合职责、重复结构、可独立复用的 UI 块、状态编排、请求编排或数据转换时，才按功能拆到子组件、hook、service/api 或 util。
2. 出现 2 处及以上重复结构时，必须优先提炼复用组件，而不是继续复制实现。
3. API 请求不能直接写在组件内，统一下沉到 `api/*.ts` 或 service 层。
4. 业务逻辑必须抽离到 hook 或 service，组件只负责 UI 组合与事件分发。
5. UI 组件必须保持无副作用，避免在渲染组件中散落请求、全局状态写入或环境依赖。
6. 业务逻辑默认保持连贯，不为了拆分而拆分；除非某段逻辑具有高通用复用价值、单个方法确实过长（例如约 80 行以上）或能隔离明确副作用 / 数据转换 / 公共契约，否则不要抽成只调用一次的微型私有方法。
7. 可读性优先于过度抽象；完整业务流程能在当前方法内一眼看到底时，优先保留在原处，减少在不同方法间频繁跳转导致的思维中断。
8. 组件架构层级不得超过 5 层，避免形成过深的包装链与调试负担。
9. 当 props 超过 3 个时，必须评估是否需要做接口抽象、上下文收敛或组件职责拆分。
10. 必须使用 TypeScript，为 props、接口数据、状态与关键函数入参/返回值补齐类型。
11. 页面或关键区域必须提供 loading、error、empty 状态，避免无反馈或空白界面。
12. 优先复用现有组件，再考虑新增组件，避免在业务侧重复封装已有能力。
13. 必须使用统一组件库（如 Ant Design 或现有 Design System），避免混用零散 UI 风格。
14. 页面标题、字段展示名、表格列头、表单 label、按钮、Tab、Tooltip、空态、校验提示和枚举显示值必须统一走 i18n；如果本地封装支持默认值，可使用中文作为默认展示文本。

## Interaction And Visual Consistency

- 交互或视觉优化前，必须先扫描相邻页面，提炼当前工作区的 Ant Design / Ant Design Vue 组件语言、间距、状态色、圆角/阴影和动效节奏。
- 如结合外部设计建议，只允许在现有框架风格和 Ant Design 体系内增强信息层级、状态反馈和微交互，不得另起视觉体系。
- 非用户明确要求时，不新增与当前工作区不一致的字体体系、品牌色、背景装饰、布局骨架或重型动效。
- `loading`、`error`、`empty`、`disabled`、`hover`、`focus`、`selected` 等状态必须与现有页面保持一致，并优先复用组件库状态能力。
- 参考设计只能辅助当前业务表达；不能因为某个案例“看起来高级”就搬入与当前业务无关的状态组织、统计卡或操作区。
- 线框、原型和设计说明中的开发者标注不得直接出现在最终用户界面中；最终页面文案必须服务终端用户理解和操作。
- 指标卡、趋势图、分布图、排行或汇总数字必须对应真实数据源、刷新方式和业务用途；三者任一不清楚时不得实现。
- 用户可见文案的语言切换、默认值和取值路径必须与相邻页面保持一致，不要在单个页面里临时拼装另一套 i18n 写法。

## Business-Driven Page Design

- 页面结构必须先服务业务任务，再服务数据实体；先判断用户是在“浏览列表”还是在“完成工作、做决策、排查问题、推进流程”。
- 对监控、告警、审批、运维、巡检、联动配置、流程推进等场景，优先考虑工作台、详情驱动、分步流、时间线、卡片概览、关系视图等结构，而不是默认回退为 CRUD 列表页。
- 只有当业务核心确实是批量检索、筛选、增删改查时，才采用标准管理页组合。
- 允许借鉴相似业务案例，但必须优先说明相似点，并对对象、状态、动作和信息层级做本业务语义映射。
- 如果一个统计数字、趋势、排行、占比、卡片或图表不能帮助用户理解状态、触发动作或做出判断，就不应出现在页面上。
- 当页面结构选择存在明显不确定性时，应先提问或先给低保真线框 / 效果预览，再进入正式实现。
- 只有在页面已被判断为标准管理页时，才应优先使用搜索区 + 表格 + 弹窗/抽屉这一套管理页组合。

## 高频事件请求控制

- 对同一事件或同一触发源的高频请求（如输入搜索、滚动加载、窗口 resize、频繁点击）必须评估并使用防抖或节流。
- 输入类、筛选类、联想搜索类请求优先用防抖；滚动监听、拖拽、持续触发类请求优先用节流。
- 禁止在高频事件中直接发起无控制请求，避免重复调用、接口抖动与页面卡顿。
- 若业务必须实时请求，需说明原因并补充并发控制策略（如取消前序请求、忽略过期响应）。

## Vue Props 语法限制

- 在 `Vue 3` 的 `script setup lang="ts"` 组件中，优先跟随当前模块相邻组件的 `props` 声明风格；若模块已有运行时对象语法约定，则不使用泛型写法 `defineProps<T>()`。
- 需要运行时默认值、校验或与相邻代码保持一致时，使用 `PropType` 绑定具体字段类型，并在 `required`、`default` 等运行时配置中显式声明。
- 错误写法：`defineProps<GaugeWidgetProps>`
- 正确写法示例：

```ts
const props = defineProps({
  info: {
    type: Object as PropType<GaugeInfo>,
    required: true
  },
  style: {
    type: [Object, String] as PropType<GaugeWidgetProps['style']>,
    default: () => ({})
  },
  isEdit: {
    type: Boolean,
    default: false
  }
})
```

## 注释规范

### 注释原则

- 注释解释“为什么”，不要重复代码已经表达清楚的“是什么”。
- 不要求通篇代码都有注释；简单赋值、普通事件转发、直观 computed、模板展示和组件 props 透传不需要注释。
- 复杂业务规则、后端兼容逻辑、跨组件状态联动、权限/平台差异、异步竞态保护、路由参数兼容、临时方案、历史包袱必须写注释。
- 设计文档中的业务规则、状态流转、异常分支等内容，必须映射到具体实现代码附近。例如某段实现“审批通过后自动生成出库单”，就在该代码段前说明这条业务规则。
- 注释应贴近实际业务代码，优先描述“这段代码在处理什么业务”，不要把业务说明堆在类头或方法头，也不要让注释和实现位置割裂。
- 注释必须短而具体，说明触发条件、业务原因、边界约束或移除条件；不要写成需求文档、交互说明或页面文案。
- 禁止空泛注释、无意义注释、重复代码表层含义的注释、与当前行为不一致的过期注释。不要写“这里进行校验”“处理数据”等无业务含义的描述；必须说明校验对象、触发条件、业务目的及影响结果。
- 注释用词要自然业务化，禁止出现“需求要求”“设计要求”“根据文档”等机械化表述，直接以业务视角描述实际逻辑。
- 临时兼容或过渡方案的注释应说明触发原因、移除条件或关联事项，避免长期遗留后无法判断是否还能删除。
- 修改代码时同步校验相邻注释是否仍然准确，避免注释和实际行为不一致。
- 修改业务逻辑时必须同步更新对应注释；如果注释无法贴近对应代码，优先拆分或移动注释，而不是保留远离实现的大段说明。

### 必须注释的典型场景

- **业务口径**：状态优先级、统计口径、隐藏/禁用条件、批量操作边界和异常分支。
- **兼容转换**：前后端字段不一致、分页基准差异、旧路由参数、历史数据结构和灰度接口。
- **状态联动**：多个 store、URL 查询参数、Tab、弹窗、轮询、订阅事件之间存在隐式同步关系。
- **竞态保护**：请求去重、延迟刷新、取消上一次请求、轮询暂停恢复和卸载清理。
- **公共契约**：公共 composable、service、api wrapper、跨页面复用组件的参数约束和副作用。

### 不需要注释的典型场景

- 变量名、函数名、类型名已经表达清楚的普通逻辑。
- 仅把用户操作转发给父组件或 hook 的事件函数。
- 模板中的普通字段展示、按钮渲染、表单项布局和组件 prop 透传。
- 与相邻代码完全一致的常规 loading、empty、error 状态处理。

### 推荐注释示例

```ts
// 后端分页从 0 开始，UI 展示从 1 开始，此处做转换
const displayPageNo = computed(() => pagination.pageIndex + 1)
```

```ts
// 运行中优先于异常展示，避免轮询延迟导致“刚启动即失败”的误导状态。
const visibleStatus = computed(() => (task.value?.running ? 'running' : task.value?.status))
```

```ts
// 兼容旧链接中的 productId，新页面统一使用 targetId；后续旧菜单下线后可移除。
const targetId = computed(() => route.query.targetId || route.query.productId)
```

```ts
// 切换项目会重建权限上下文，必须清空已选项，避免把上一项目的资源提交到当前项目。
watch(projectId, () => selectedRowKeys.value = [])
```

```ts
/**
 * 生成资产详情页查询条件。
 *
 * routeQuery 来自可分享链接，字段可能是 string 或 string[]；返回值只保留接口支持的精确匹配条件。
 */
function buildAssetQuery(routeQuery: LocationQuery): AssetQueryParams {
  // ...
}
```

### 禁止注释示例

```ts
// 设置 loading 为 true
loading.value = true
```

```ts
// 点击确定
const onConfirm = () => emit('confirm')
```

```ts
// 定义 columns
const columns = computed(() => getColumns())
```

### JSDoc

- 复杂函数、公共函数、公共 composable、API 封装建议使用 JSDoc。
- JSDoc 应优先说明业务意图、关键参数约束、返回结构和异常行为，不要把函数名或类型签名再复述一遍。

```ts
/**
 * 获取视觉设备列表
 *
 * @param params - 查询参数
 * @returns 分页结果，含 data 数组和 total 总数
 * @throws {ApiError} 当接口返回 code !== 0 时抛出
 */
async function fetchVisionList(params: VisionListParams): Promise<PageResult<VisionVO>> {
  // ...
}
```

## 禁止模式

- 提交调试日志、注释掉的废弃代码、临时代码块。
- 在组件中硬编码后端路径、权限规则或环境配置。
- 使用 `as any` 或大范围 `any` 直接绕过类型错误。
- 同一业务模型在多处重复且冲突定义。
- 在已有运行时对象语法约定的模块中改用泛型 `defineProps<T>()` 声明组件 props。
- 以“美化”为名混入脱离当前框架的自创风格、全新设计语言或不兼容的交互动线。
- 未经业务分析就把需求直接落成老套后台管理 CRUD 模板。
- 为了页面显得“完整”而添加无业务意义的统计数字、装饰卡、假图表或无来源数据。
- 在数据源、刷新逻辑或业务用途未确认前先做统计卡、图表或排行。
- 直接照搬完全不同业务场景中的信息架构、指标体系或操作路径。
- 在页面组件、schema、columns、按钮配置或状态映射中散落硬编码用户可见文案，而不走统一 i18n。
- 把“交互方式”“设计原理”“模块说明”“待接接口”等原型或开发说明直接展示给终端用户。
- 用注释重复代码表层含义，或保留与当前代码行为不一致的过期注释。

## 交付前检查建议

1. 执行目标项目构建或类型检查命令（如 `pnpm build`、`pnpm typecheck`）。
2. 对关键链路做冒烟验证：查询、保存、删除、路由跳转、权限按钮。
3. 若改动影响通用能力，补充最小可复现验证路径。
4. 说明未执行的检查项及原因（如环境限制、依赖缺失）。

## 自检清单

- 是否避免了重复实现并复用了既有能力。
- 是否补齐了关键入参、出参与状态类型。
- 是否收敛了 `any` 的范围并避免无边界断言。
- 是否在现有设计体系内完成了交互优化，而不是额外创造一套页面风格。
- 是否确认了页面结构真正匹配业务任务，而不是为了省事落成 CRUD 套壳。
- 是否避免了无意义的数据展示，并确保每个数字、图表、卡片或快捷入口都有真实业务价值。
- 是否避免把原型说明和开发注释泄漏到最终界面。
- 是否避免写死用户可见字段展示名、列头、按钮和提示文案，并统一走了当前工作区 i18n。
- 是否只在解释业务原因、兼容背景、边界约束或公共 API 契约时补充注释，并确保注释与代码行为一致。
- 若本次前端改动没有新增任何注释，是否已确认改动只包含直观展示、简单赋值、事件转发或常规状态处理，不存在必须注释的关键逻辑。
- 是否验证了关键交互与权限行为。

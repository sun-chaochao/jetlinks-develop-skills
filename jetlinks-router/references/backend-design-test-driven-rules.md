# 后端设计与测试驱动规则

本文件用于约束 JetLinks 后端较大改动或新功能开发：先形成可确认的设计稿和测试目标，再进入实现，最后以真实业务场景和数据完成验证。

## 触发条件

命中以下任一条件时，必须进入设计与测试驱动流程：

- 新增后端能力、接口、模块、协议接入、事件链路、批处理任务或跨模块调用。
- 修改 Entity / Service / Controller / Repository / Command / Event / Subscribe / Codec 等核心行为，且会影响用户可见结果、持久化数据、消息、权限、状态流转或对外 API。
- 改动跨越两层以上架构边界，例如 Controller + Service + Repository、CRUD + Event、协议 + 物模型、命令服务 + 响应式链路。
- 需求存在多个合理方案、兼容性风险、数据迁移风险、并发或事务风险、性能风险，或用户描述仍在变化。
- 修复 bug 但需要重新定义业务语义、异常语义、边界行为或回归验证方式。

可豁免场景：

- 单个字段文案、导入、注解、格式、注释或配置的小修。
- 已有设计文档明确覆盖本次改动，且本次只按原设计补齐实现；仍需先列出测试目标。

## 设计稿落点

设计稿必须落到当前工作区对应文档目录，而不是只写在对话里。落点选择先遵循 [`document-placement-rules.md`](document-placement-rules.md)：README 只放长期总览，不放单次设计过程、测试报告或 PR 证据；先更新已有归属文档，避免为每个任务新增碎片。

落点选择顺序：

1. 读取仓库根 README、AGENTS、已有 docs 索引或模块文档规则，确认文档归属；不要把 README 当成任务文档。
2. 如果已有功能设计、ADR、模块设计或计划文档覆盖同一主题，优先更新该文档。
3. 优先使用受影响模块已有的 `docs/`、`docs/plans/`、`docs/design/`、`docs/adr/` 或等价目录。
4. 如果仓库根已有 `docs/plans/`、`docs/design/`、`docs/adr/`，优先落到最贴近用途的根目录。
5. 如果没有任何文档目录，但用户已经要求较大后端设计先行，默认创建 `docs/plans/`；若仓库明确禁止新增文档目录，先向用户说明阻塞并确认替代落点。

文件名使用稳定、可检索的形式：

```text
docs/plans/<yyyy-mm-dd>-<feature-slug>.md
```

如果模块已有命名规范，跟随模块规范。`feature-slug` 使用业务对象或能力名，不使用 `temp`、`todo`、`note`、`summary` 这类泛名。

避免碎片化：

- 同一功能只保留一个主设计稿，设计、任务拆分和测试目标放在同一文档内。
- 子任务进度、测试命令、覆盖率和 CI 结果优先放 PR 描述或 CI，不为每次运行新增测试报告文档。
- 只有架构决策、API 说明、模块职责或可复用经验已经稳定时，才同步到对应长期文档。

## 设计稿内容

设计稿至少包含：

- 背景：当前问题、触发场景、已有实现事实。
- 目标：本次要达成的业务能力和可观察结果。
- 非目标：明确不做的范围，防止顺手扩大。
- 影响范围：模块、接口、数据结构、权限、i18n、事件、协议、缓存、定时任务、MBean / 运维操作、外部依赖。
- 方案：推荐方案、关键取舍、被拒绝方案及原因。
- 数据与边界：实体字段、查询条件、AssetsHolder 数据权限、事务边界、事件时机、消息幂等、跨模块调用方式、SQL 数据库兼容性、链路追踪与关键埋点、MBean 运维可观测性、兼容性或迁移策略。
- 任务拆分：可执行步骤，按先测试目标、再实现、再验证组织。
- 测试目标：见下文，必须映射到真实使用场景和数据。
- 风险与待确认：需要用户确认的业务语义、数据口径、权限边界、性能或兼容性风险。

## 链路追踪与可观测性设计

新增后端能力、关键状态流转、命令执行、规则场景、协议链路、批处理、跨模块调用、事件 / 订阅或外部 I/O 时，设计稿必须说明是否需要 TraceHolder 手动埋点。

需要埋点时写清：

- 关键 span：业务阶段和稳定 span 名称。
- 关键属性：业务 ID、动作、状态、结果、数量、分支等非敏感信息。
- 上下文传播：HTTP、RSocket、消息、事件、RuleData、命令 metadata 等边界如何传递 trace context。
- 粒度控制：批量或高频链路按批次 / 阶段记录，不为每条普通数据创建噪声 span。
- 安全边界：不记录 token、密钥、完整 Authorization / Cookie、隐私字段、完整大 payload 或无界集合。

如果不需要新增埋点，设计稿写明原因，例如已有通用 filter / SQL executor / command codec / 领域 tracer 覆盖，或本次只是纯规则 / 文档 / 小范围重命名。

## MBean 与运维可观测性设计

涉及常驻内存任务、缓存、队列、buffer、重试池、死信队列、会话管理器、连接管理器、订阅管理器、协议支持管理器或后台执行器时，设计稿必须说明是否需要提供 MBean 来辅助运维定位问题。

需要 MBean 时写清：

- MBean 名称：稳定的 `org.jetlinks:type=...,name=...`，动态 name 要转义。
- 统计指标：大小、积压、死信、命中 / 未命中、输入 / 输出、成功 / 失败、丢弃数、耗时分布、最近错误、最后刷新时间等。
- 运维操作：刷新缓存、flush、compact、重置统计、手动重试、清理过期、查看有限样本等。
- 生命周期：启动注册、关闭注销，重复注册或注册失败不影响主业务。
- 安全边界：不暴露 token、密钥、完整 payload、隐私字段、无界集合；操作参数有上限，影响业务的操作要可解释、可验证。

如果不需要新增 MBean，设计稿写明原因，例如已有 `PersistenceBuffer`、`ClusterEventBus`、`LocalTracingService`、`NetworkMbeanSupport` 等同类监控覆盖，或本次能力无常驻状态。

## SQL 可移植性与性能设计

涉及复杂 SQL、原生 SQL、聚合、多表关联、深分页或批量写入时，设计稿必须说明：

- 优先方案是否能用 `QueryHelper`、CRUD DSL、QueryParam 或当前模块封装表达。
- SQL 是否使用标准 SQL 和通用函数；除非用户明确要求或模块已限定数据库，不默认使用特定数据库方言。
- 如果必须使用数据库方言，写清数据库范围、使用原因、不可移植风险、替代方案、回退方式，并列为用户确认项。
- 性能预估：真实数据量、筛选组合、排序字段、分页深度、索引命中、并发访问、批量大小和可能的锁 / 扫表风险。
- 压力测试目标：数据规模、并发数、查询 / 写入频率、耗时阈值、资源指标和通过标准；无法压测时写明替代验证与剩余风险。

## 发布边界与兼容性判断

兼容性判断是通用发布边界规则，不只适用于 CRUD。设计时先区分“已发布事实”和“PR 内中间形态”：

- 已合入主线、已发布到生产、已有外部调用方、已有持久化数据、缓存、消息、配置、保存查询、外部链接，或已公开 API / DTO / Event / Topic / Command / 协议报文 / 前端路由参数 / QueryParam / `termType` 依赖时，才按兼容性风险处理，设计迁移、适配、灰度或过渡策略。
- 同一个 PR 内尚未发布的旧逻辑、旧字段、旧接口、旧 DTO、旧事件结构、旧 Topic、旧 Command、旧协议报文、旧配置项、旧路由参数、旧 `termType` 或旧测试期望，优先直接收敛到最终最佳实践，并同步修改同 PR 内调用方、测试和文档，不为了兼容 PR 中间形态保留多套协议、分支代码、适配器或迁移逻辑。
- 如果不确定是否已有外部依赖，先把“是否已发布 / 是否已有持久化或调用方依赖”列为待确认项询问用户。
- PR 描述中可说明“不涉及已发布兼容性；本次为 PR 内未发布逻辑收敛”，但不要因此省略真实测试。

智能体执行约束：

- 设计稿中如果出现“兼容、旧字段、旧接口、旧参数、旧事件、旧协议、旧配置、旧路由、旧测试期望”，必须写出兼容对象来自哪里：已发布 / 已持久化 / 外部依赖 / 同 PR 中间形态 / 不确定。
- 结论为“同 PR 中间形态”时，推荐方案必须是单一 canonical 实现，并把删除旧分支、旧别名、旧适配、旧测试期望列入任务；不要给出“双写一段时间”“新旧都支持”“保留兜底解析”这类方案。
- 结论为“不确定”时，只问一个具体问题确认发布或外部依赖状态；确认前不要把兼容代码写进方案。
- 结论为“已发布 / 已持久化 / 外部依赖”时，才设计兼容、迁移、灰度、回滚或清理计划，并写清退出条件。

## 用户确认门槛

设计稿写入文档后，必须向用户汇报设计稿路径、核心方案、任务拆分和测试目标，并等待明确确认再开发。

明确确认包括：

- “确认”
- “按这个做”
- “继续实现”
- “采用方案 A”

不能把沉默、模糊认可或仍带疑问的反馈当成确认。如果用户要求调整设计，先更新设计稿，再重新确认。

## 测试目标先行

开发前必须先从设计稿中提取测试目标。测试目标要说明：

- 真实业务场景：谁在什么状态下做什么，期望系统产生什么结果。
- 真实数据形态：使用现有 fixture、样例报文、相邻模块测试数据、协议文档样例或最接近生产约束的数据；不能只用 `foo` / `bar` / `test` 占位。
- 正常路径：核心能力按预期工作。
- 异常路径：权限不足、资源不存在、参数非法、状态不允许、重复请求、外部依赖失败等关键分支。
- 回归路径：本次修复或变更最容易重新出错的路径。
- 边界路径：空集合、大批量、分页边界、重复数据、并发或幂等、协议粘拆包、消息乱序等，按实际风险选择。
- 数据权限路径：涉及 CRUD 时，按 [`../../jetlinks-assets-permission/SKILL.md`](../../jetlinks-assets-permission/SKILL.md) 走统一 AssetsHolder 资产权限；覆盖不同 AssetsHolder 绑定、资产类型、权限动作、关联资产字段下的可见、不可见、可操作和不可操作场景；如果资产类型、关联字段、权限动作或例外规则不确定，先把待确认项写入设计稿并询问用户。
- SQL 性能路径：涉及复杂 SQL、原生 SQL、聚合、多表关联、深分页或批量写入时，必须包含标准 SQL 可移植性判断、索引 / 扫表风险、真实业务数据规模和压力测试或替代性能验证。
- 链路追踪路径：涉及关键后端业务链路时，必须包含 TraceHolder / MonoTracer / FluxTracer 埋点目标、关键属性、上下文传播和敏感信息排除；如果已有平台自动追踪覆盖，说明不新增手工埋点的依据。
- MBean 运维路径：涉及常驻内存、缓存、队列、重试或后台执行器时，必须包含 MBean 决策、统计指标、监控字段、安全内部操作、生命周期和敏感信息排除；如果已有 MBean 覆盖，说明复用依据。
- 集成触发判断：是否涉及数据库、消息、事件、协议、跨模块边界、外部依赖或启动装配；触发则补集成测试或说明受阻原因和替代验证。
- 验收标准：每个测试目标对应可执行命令、预期断言和通过标准。

## 开发与验证顺序

推荐顺序：

1. 写入或更新设计稿。
2. 等待用户确认。
3. 根据设计稿制定测试目标，优先新增或更新测试。
4. 实现生产代码，保持改动范围与设计稿一致。
5. 持续运行相关测试，直到测试目标达成。
6. 如果实现中发现设计假设不成立，先回写设计稿并让用户确认变更，再继续。
7. 交付前把最终测试命令、结果、覆盖率、集成测试结果或不适用原因写入 PR 描述或 CI 报告；设计稿只保留必要的验证摘要，README 不记录测试报告。

## 测试真实性约束

测试目标不是为了“让测试变绿”，而是为了证明真实业务场景可靠。

禁止：

- 删除测试、注释断言、降低有效断言、只保留 `notNull` 或空跑测试。
- 把期望值改成当前错误实现的输出，绕过真实业务语义。
- 只 mock 掉被测核心逻辑，导致测试没有验证真实分支。
- 用宽泛 `any()` / `lenient()` 掩盖关键参数、权限、状态或消息内容。
- 用随机 sleep、扩大超时、吞异常、忽略异步结果来让测试偶然通过。
- 只运行无关模块测试，或用无关覆盖率冒充本次改动覆盖率。
- 为了测试方便改变生产代码语义、放宽权限、绕过校验或新增测试专用分支。

要求：

- 断言业务结果、持久化结果、事件内容、对外响应、异常类型与 i18n code、权限行为、协议报文或消息字段中的关键事实。
- 测试数据必须能解释为什么代表真实场景；如果使用合成数据，需要满足生产约束和边界条件。
- 外部依赖不可用时，用契约测试、适配器测试、记录的真实报文或模块内已有样例替代，并明确剩余风险。
- 测试失败时先分析根因；如果根因是设计目标不合理，更新设计稿并让用户确认，不要偷偷降低测试目标。

# JetLinks 实时订阅规则

本文件用于处理 Topic、消息流和实时事件消费。

## 何时使用

- 当前需求是持续消费而不是一次性请求
- 需要 `@Subscribe`、EventBus、消息总线或系统 Topic
- 需要处理设备消息、广播消息、系统事件流

## 核心原则

1. Topic 必须来源于当前工作区的真实约定
   - 先查现有订阅者、常量、Topic 工具类、事件定义。

2. 订阅处理器要保持幂等
   - 消息重放、重复投递、共享消费在实时场景里很常见。

3. 高吞吐场景要考虑缓冲与批量
   - 如果当前模块已有 `buffer`、`window`、批处理 sink、去重策略，优先复用。

4. 不把一次性查询伪装成订阅
   - 订阅是流式消费，不是远程查询替代品。

## 设计顺序

1. 确认消息来源
   - 设备消息、系统消息、领域广播、业务广播。

2. 确认 Topic 形状
   - 从现有代码中复制最接近的 Topic 模式。

3. 确认订阅特性
   - 本地、broker、shared、优先级、缓冲策略等，以现有模块约定为准。

4. 确认输出动作
   - 持久化、转发、聚合、同步缓存、触发事件。

## 推荐做法

- 返回 `Mono<Void>` 或当前框架惯用的异步类型
- 在处理器内部保持非阻塞链路
- 对脉冲型高频消息做去重、批量或降噪
- 把复杂业务拆到服务层，不把整个流程都塞进订阅方法

## 注释落地

- Topic 兼容、共享订阅语义、消息重放幂等、去重窗口、buffer / window 边界、失败重试、订阅释放或上下文传播如果不是显而易见，必须在订阅处理器或 helper 方法旁边写短注释。
- 注释说明业务边界、协议 / Topic 来源、重放或批量策略；不要逐行解释 Reactor 操作符。
- 只做简单转发且方法名已经说明语义时，可不写实现注释。

## 何时联动其他规则

- 订阅后需要保存或查询实体：继续读取 `advanced-crud-rules.md`
- 订阅后需要解耦副作用：继续读取 `event-driven-rules.md`

## 自检清单

- Topic 是否来自真实代码或常量
- 订阅特性是否与现有模块一致
- 处理器是否幂等
- 是否避免了阻塞和逐条低效操作
- 是否为 Topic 兼容、重放幂等、批量 / 去重边界或订阅生命周期补了必要代码注释

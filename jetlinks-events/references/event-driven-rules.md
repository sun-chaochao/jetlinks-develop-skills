# JetLinks 事件驱动规则

本文件用于处理实体生命周期事件、Spring 事件和领域联动。

## 何时使用

- 保存、修改、删除后需要联动其他逻辑
- 需要把副作用从主流程中拆出去
- 需要在事务提交后触发异步动作

## 核心原则

1. 事件负责解耦，不负责替代主业务
   - 主业务仍应在 Service/Controller 中保持清晰。

2. 先选正确的触发时机
   - 创建前、保存前、修改后、删除后、提交后，不要混用。

3. 跟随当前模块的执行模型
   - 响应式模块优先沿用异步事件处理方式。
   - 阻塞式模块跟随同步事务边界。

4. 外部副作用优先在提交后执行
   - 发布消息、远程调用、缓存同步、关系清理等，优先避免事务未提交就外泄。

5. 低上下文模板仓库下，副作用默认优先事件驱动
   - 当需求是“保存后同步”“删除后清理”“修改后联动”时，默认先考虑事件监听，而不是把逻辑直接写进 CRUD 主流程。

## 选择事件类型

### 使用实体生命周期事件

适用：
- CRUD 后的数据联动
- 冗余字段同步
- 关联数据清理

### 使用自定义领域事件

适用：
- 业务语义明显独立
- 一个动作需要被多个模块消费

## 低上下文模板仓库默认策略

如果当前仓库还没有成熟事件样例：

1. 先用实体生命周期事件起步
   - 它比自定义领域事件更容易保持最小实现。

2. 只有在业务语义已经明显独立时，再抽自定义领域事件

3. 如果用户没有要求同步返回结果
   - 默认将副作用从主流程中拆到事件处理器

## 实现要求

1. 处理器职责单一
   - 一个处理器只做一类联动。

2. 修改后逻辑读取“修改后”数据
   - 如需前后对比，再同时读取 before/after。

3. 注意防止循环触发
   - 事件处理里再次更新同一实体时，要确认是否需要抑制二次事件。

4. 大批量联动要批处理
   - 不逐条串行做低效操作。

## 注释落地

- 事件触发时机、事务提交后执行、幂等策略、循环触发抑制、批量上限、失败重试或跨模块副作用不是一眼能看懂时，必须在处理器代码旁边写短注释。
- 注释说明“为什么放在这个时机 / 为什么这样防重 / 为什么批量边界是这个值”，不要解释“监听事件并调用方法”。
- 简单的一步事件转发或命名清晰的处理器可以不写实现注释，但新增公共 Event / Listener / Handler 类型仍按公共契约补类注释。

## 何时跳转其他规则

- 事件里需要批量更新或复杂查询：继续读取 `advanced-crud-rules.md`
- 事件里要调用其他边界能力：继续读取 `cross-service-call-rules.md`

## 自检清单

- 触发时机是否正确
- 是否考虑事务提交时机
- 是否避免循环触发
- 是否保持了处理器单一职责
- 是否为非显而易见的触发时机、幂等、循环抑制、批量边界或副作用边界补了必要代码注释

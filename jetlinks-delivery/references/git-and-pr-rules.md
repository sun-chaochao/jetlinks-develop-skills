# Git、分支、PR 与测试交付规则

当任务不仅涉及“写代码”，还涉及“提交代码、准备交付、发起 PR、说明测试结果”时，读取本文件。

如果任务重点是中文 commit message 起草、合规审查或 shell 提交命令，继续读取：

- [`commit-message-zh.md`](commit-message-zh.md)
- [`shell-commit-examples.md`](shell-commit-examples.md)

## 目标

确保 JetLinks 项目的交付过程符合以下要求：

- 提交标题对齐仓库既有风格
- 禁止直接 push 到主干或集成版本分支
- 通过 PR 合并
- 后端新增功能或既有功能变动必须补齐对应单元测试
- 较大后端改动或新功能必须先有已确认的设计稿、任务拆分和测试目标
- 提交前完成相关单元测试，并按触发条件完成集成测试
- 交付前确认必要文档已与实现保持一致
- 复杂 SQL 或原生 SQL 需要说明数据库兼容性、方言风险和真实性能验证
- 复杂逻辑、公共类和 SPI 扩展点需要说明注释 / 公共契约结论
- 关键后端业务链路需要说明 TraceHolder 链路追踪埋点决策
- 常驻内存任务、缓存、队列或后台执行器需要说明 MBean 运维可观测性决策
- 在 PR 描述中用数据说明测试结果和覆盖率

## 提交标题规范

优先对齐仓库现有历史，默认采用：

```text
type(scope): summary
```

约束：

- `type` 与仓库历史保持一致，优先使用 `feat`、`fix`、`refactor`、`docs`、`test`。
- `scope` 选择本次改动的真实模块、业务域或规则域。
- `summary` 使用简洁中文动宾短语，直接说明结果。
- 除非历史明确允许，否则不要省略 `scope`。

参考仓库历史风格：

- `refactor(prompt): 扩展低上下文边界决策规则`
- `refactor(基础模块): 优化菜单逻辑`
- `refactor(设备管理): 优化实体拓展型`

避免：

- `update`
- `fix bug`
- `misc`
- `test`
- 无法体现模块和结果的空泛标题

如需进一步决定 `type`、`scope`、`body`、`footer` 或 ready-to-run 命令形式，转到
[`commit-message-zh.md`](commit-message-zh.md) 和
[`shell-commit-examples.md`](shell-commit-examples.md)。

## 分支与合并策略

禁止直接 push 到以下分支：

- `master`
- `main`
- 版本集成分支，例如 `2.11`、`2.12`
- 其他被团队视为主线或发布线的长期分支

标准流程：

1. 确认目标合入分支，例如 `2.12`。
2. 从该基线分支拉出临时开发分支。
3. 在临时分支上完成修改、测试和提交。
4. push 临时分支到远端。
5. 创建 PR，base 指向目标版本分支。

如果用户要求“直接推到 2.12”这类操作，应改为：

- 说明该分支属于受保护主线，不直接 push。
- 改走“临时分支 + PR”流程。

## GitHub PR 创建方式

如果当前仓库远端是 GitHub，且用户要求“发 PR”“提交 PR”“推到 GitHub 并开 PR”，默认推荐使用 `gh` 创建 PR，而不是只给一个网页链接让用户手动补完。

推荐顺序：

1. 用 `gh auth status` 检查 GitHub CLI 是否已登录。
2. 用 `gh repo view` 确认仓库、默认分支和目标 base。
3. 在创建 PR 之前，先确认用户是否认为当前任务已经完成。
4. 如果任务未完成、仍需继续开发、仍有待补测试或待确认事项，创建 draft PR。
5. 如果任务已完成，且用户确认可以进入评审阶段，再创建 ready for review 的 PR。
6. 在临时分支 push 完成后，优先使用 `gh pr create` 创建 PR。

带沙箱的 agent 环境额外约束：

- 如果当前工具支持非沙箱审批或提权机制，`gh` 相关命令不要在沙箱内执行。
- 包括 `gh auth status`、`gh repo view`、`gh pr create` 在内，都应通过当前工具支持的非沙箱机制执行。
- 如果当前工具不支持非沙箱执行，直接说明环境限制，再按可用权限尝试或降级为提供可点击 PR 链接。
- 如果认证失效，先明确阻塞点，让用户完成 `gh auth login` 后再继续。

状态选择约束：

- 不要替用户假定“已经完成，可以直接 review”。
- 只有用户明确确认已完成，或上下文中已清楚表明“请直接提评审 PR”，才创建 ready for review。
- 如果上下文不明确，默认先问；只有确认未完成时，才主动选择 draft。

允许的降级方式：

- 如果 `gh` 未安装，先说明阻塞原因。
- 如果用户不希望 agent 直接创建 PR，再退回到提供可点击的 PR 链接。
- 只有在 `gh` 不可用或用户明确要求手动处理时，才不使用 `gh pr create`。

## PR 大小控制

默认不建议在一个 PR 中提交大量代码，尤其不要把多个独立主题、重构、功能开发、测试修复和文档整理混在一起。

推荐做法：

1. 一个 PR 只解决一个清晰的问题或一组强相关改动。
2. 如果改动同时包含“功能实现 + 大范围重构 + 测试修复 + 文档整理”，优先拆成多个 PR。
3. 如果工作树里存在明显无关文件，不要一次性全部提交。
4. 如果确实无法拆分，也要在 PR 描述中明确说明为什么必须合并提交。

不推荐：

- “顺手一起改掉”的跨主题提交
- 超大 diff 且缺乏清晰边界的 PR
- 把用户尚未确认完成的多个阶段性成果一次性塞进 review PR

## PR 内未发布变更的兼容性

兼容性判断不只适用于 CRUD。同一个 PR 内调整尚未合入主线或发布生产的逻辑时，优先把 PR 收敛成最终最佳实现，而不是为了兼容中间提交保留额外代码。

优先收敛的场景：

- 旧字段、旧接口、旧 DTO、旧事件结构、旧 Topic、旧 Command、旧协议报文、旧配置项、旧前端路由参数、旧 QueryParam、旧 `termType` 或旧测试期望只存在于当前 PR 的早期实现。
- 调用方也在同一个 PR 中，可以同步修改。
- 没有已保存查询、持久化数据、外部系统、生产配置或已发布 API 依赖旧形态。

需要保留兼容或迁移的场景：

- 旧逻辑已经合入主线、发布生产或被其他分支 / 外部调用方使用。
- 已有数据库数据、缓存、消息、配置、保存查询、Topic、Command、API 契约、协议报文、前端外部链接或用户已保存状态依赖旧形态。
- 用户明确要求兼容历史行为。

交付要求：

- 交付前主动搜索本次改动中的兼容、fallback、deprecated、legacy、旧字段、旧接口、旧参数、旧事件、旧配置等痕迹；凡是没有已发布、持久化或外部依赖依据的兼容代码，都必须在进入 ready for review 前删除。
- 如果是 PR 内未发布逻辑收敛，PR 描述中说明“不涉及已发布兼容性”即可，不要新增无意义兼容分支。
- 如果需要兼容，写清兼容对象、退出条件、迁移方式和后续清理计划。

## 后端设计与测试驱动门禁

较大的后端改动或新功能进入交付前，必须确认已执行设计与测试驱动流程。触发条件和设计稿要求以
[`../../jetlinks-router/references/backend-design-test-driven-rules.md`](../../jetlinks-router/references/backend-design-test-driven-rules.md)
为准。

交付前必须核对：

- 设计稿已写入当前工作区对应文档目录，而不是只停留在对话里。
- 设计稿包含目标、非目标、影响范围、方案取舍、任务拆分、测试目标、风险和待确认项。
- 用户已明确确认设计稿；如果用户要求调整，已先更新设计稿再开发。
- 测试目标先于实现制定，并覆盖真实使用场景、真实数据形态、正常路径、异常路径、回归路径和必要边界。
- 涉及 CRUD 时，已按 `$jetlinks-assets-permission` 分析 AssetsHolder 数据权限控制：资产类型、权限动作、查询注入、详情访问、更新 / 删除 / 批量操作校验、关联资产边界；拿不准的资产类型、关联字段、权限动作或例外规则已询问用户。
- 涉及复杂 SQL、原生 SQL、聚合、多表关联、深分页或批量写入时，已确认优先使用 QueryHelper / CRUD DSL / 标准 SQL；如使用特定数据库方言，已在设计稿中写明用户要求、数据库范围、不可移植风险和替代方案。
- SQL 性能目标已覆盖真实数据量、筛选组合、排序字段、分页深度、索引命中、并发访问和批量规模；需要压测时已有压力测试目标。
- 涉及复杂业务、公共类、SPI / 扩展点、Provider、Codec、Listener 或 CommandHandler 时，已确认注释边界：类注释和 SPI 方法注释完整，必要 `@since` / `@see` 已指向真实版本、订阅相关类型、默认实现或参考实现，普通实现注释保持简洁。
- 涉及关键后端业务链路、状态流转、命令执行、事件 / 订阅、协议链路、批处理或外部 I/O 时，已确认 TraceHolder / MonoTracer / FluxTracer 埋点目标、关键属性、上下文传播和敏感信息排除；不新增埋点时有明确理由。
- 涉及常驻内存任务、缓存、队列、buffer、重试池、会话 / 连接 / 订阅管理器或后台执行器时，已确认 MBean 决策、统计指标、监控字段、安全内部操作、生命周期和敏感信息排除；不新增 MBean 时有明确理由。
- 实现中若发现设计假设不成立，已回写设计稿并重新确认。

门禁约束：

- 缺设计稿或缺用户确认的较大后端改动，不创建 ready for review PR。
- 发现仅为了同 PR 未发布中间形态而保留的兼容分支、旧别名、双解析、迁移代码或旧行为测试时，不创建 ready for review PR；先收敛到最终最佳实践。
- 复杂 SQL / 原生 SQL 使用特定数据库方言但未在文档和 PR 中标明风险时，不创建 ready for review PR。
- 复杂 SQL、原生 SQL、聚合、多表关联、深分页或批量写入缺少压力测试或替代性能证据时，不创建 ready for review PR；确实无法压测时只能明确阻塞或剩余风险。
- 公共类、SPI / 扩展点方法缺少完整注释，或 SPI 必要 `@since` / `@see` 版本和关联类型未确认时，不创建 ready for review PR；先补注释或写明阻塞。
- 关键后端业务链路缺少链路追踪决策时，不创建 ready for review PR；必须说明已补 TraceHolder 埋点，或说明已有平台自动追踪覆盖 / 本次不适用的依据。
- 常驻内存任务、缓存、队列或后台执行器缺少 MBean 运维可观测性决策时，不创建 ready for review PR；必须说明已补 MBean，或说明已有 MBean / 监控覆盖 / 本次不适用的依据。
- 缺测试目标或测试目标没有映射真实业务场景的数据，只能作为 draft 或阻塞说明。
- 不允许为了达成门禁而删除测试、弱化断言、只 mock 被测核心逻辑、只跑无关模块测试或把期望值改成错误实现的输出。

## 后端功能变更测试门禁

当 PR 包含后端新增功能或既有功能变动时，测试不是可选项。触发场景包括但不限于：

- 新增或调整 Controller / Service / Entity / Repository / Command / Event / Subscribe / Codec 等后端行为。
- 新增接口、查询条件、权限、字典、状态流转、数据写入、副作用或跨模块调用。
- 修改协议解析、消息编解码、订阅消费、定时任务、批处理、缓存同步或异常分支。
- 修复 bug 时改变了用户可见结果、持久化数据、事件消息或对外 API 语义。

PR 前必须完成：

- 补充或更新与改动点对应的单元测试，覆盖正常路径、关键异常路径和本次修复的回归场景。
- 涉及数据库、消息、事件、协议联调、跨模块边界、外部依赖或启动装配时，必须补充集成测试或给出现有集成测试结果。
- 不触发上述集成测试条件时，PR 中必须写清不适用原因，而不是留空。
- 收集单元测试覆盖率数据，优先给出本模块、本次 changed files / changed classes 或关键包的 line / branch 覆盖率。
- 涉及复杂 SQL、原生 SQL、聚合、多表关联、深分页或批量写入时，提供压力测试结果或等价性能验证；如用户明确要求特定数据库，PR 中标明数据库范围和方言风险。
- 在 PR 描述中列出新增或更新的测试类、核心用例覆盖点、单元测试覆盖率、集成测试结果或不适用原因。

门禁约束：

- 没有对应单元测试的后端新增功能或行为变动，不进入 ready for review。
- 触发集成测试条件但无法执行时，只能创建 draft PR 或明确标为阻塞；必须写清缺失环境、替代验证和剩余风险。
- 未触发集成测试条件时，`集成测试` 字段必须写“不适用”及原因。
- 覆盖率工具缺失时，不能省略说明；需写出缺失原因、已执行的替代验证以及人工判断的覆盖边界。
- 不允许通过删除测试、注释断言、降低有效断言或只跑无关模块测试来满足门禁。

## 测试要求

提交前必须执行自动化测试，至少覆盖本次改动涉及的核心路径。

优先级：

1. 后端新增功能或既有功能变动，先补或更新对应单元测试。
2. 先跑与改动模块直接相关的单元测试。
3. 如果改动跨模块、跨边界、跨服务、涉及数据库、消息、协议、外部依赖、启动装配或事件链路，再补集成测试。
4. 如果仓库已有覆盖率门槛，必须满足门槛后再准备 PR。

最低交付要求：

- 给出测试命令
- 给出新增或更新的测试类和核心覆盖点
- 给出通过数、失败数、跳过数
- 给出集成测试结果；未触发集成测试条件时说明为什么不适用
- 给出覆盖率结果，或说明项目缺乏覆盖率工具时的替代证据

可接受的证据示例：

- `mvn -pl modules/device-manager -am test`
- `mvn -pl modules/device-manager -am verify -P integration-test`
- surefire / failsafe 报告中的 passed、failed、skipped
- jacoco 报告中的 line / branch 覆盖率
- changed files / changed classes 的 line / branch 覆盖率

不合格的说法：

- “已测试”
- “本地没问题”
- “理论上可用”

如果测试受阻：

- 明确阻塞原因，例如环境依赖、外部服务、数据库、消息中间件
- 明确已完成的替代验证
- 明确仍然存在的风险
- 未满足团队要求前，不把该提交描述为可合并状态

## 文档一致性要求

当本次改动涉及代码、配置、接口、数据结构、页面交互、枚举/字典或业务规则变化时，交付前还需要检查对应文档是否应同步更新。

文档落点先遵循 [`../../jetlinks-router/references/document-placement-rules.md`](../../jetlinks-router/references/document-placement-rules.md)。尤其注意：README 只作为仓库或模块的长期总览，不放单次测试报告、PR 描述、临时计划、排查流水或任务总结。

优先级：

1. 先判断文档类型：长期总览、功能 / API 说明、设计 / ADR、测试证据、经验沉淀，分别放到对应归属位置。
2. 先更新已有原始文档，例如模块 README、功能说明、接口说明、仓库 `AGENTS.md`、既有 design / ADR / docs 索引；README 只同步长期稳定的模块职责、入口和链接。
3. 测试命令、覆盖率、集成测试结果等证据优先放 PR 描述或 CI 报告，不新增长期测试报告文档。
4. 只有现有文档无法承载，且内容有长期复用价值和明确维护归属时，才新增独立专题文档。

约束：

- 不要默认为了交付“留痕”而为每个任务新建 task doc、active/archive 流水或归档记录。
- 不要为了说明本次 PR 通过了哪些测试而修改 README。
- 不要把同一功能拆成多个 plan、summary、test-report、worklog 文档；较大后端改动优先用一个主设计稿承载设计、任务拆分和测试目标。
- 如果当前仓库规则要求同步文档，且文档明显滞后于实现，不要把交付描述为已完成。
- 如果用户明确要求暂不补文档，或当前任务只是纯文档 / 纯评审 / 纯同步操作，需要在 PR 描述中写清原因与风险。

## PR 描述要求

PR 描述必须是结果导向，不写空话。

至少包含：

1. 目的
2. 核心变动
3. 测试结果
4. 文档同步情况
5. 风险与未覆盖项

较大后端改动或新功能还必须包含：

- 设计稿路径
- 用户确认状态
- 测试目标达成情况
- CRUD / AssetsHolder 数据权限分析结论（如适用）
- 数据库兼容性与 SQL 性能结论（如适用）
- 注释 / 公共契约结论（如适用）
- 链路追踪 / 可观测性结论（如适用）
- MBean / 运维可观测性结论（如适用）

推荐模板：

```md
## 目的

- 修复 / 优化 / 新增什么能力
- 解决什么业务或技术问题

## 核心变动

- 模块 A：改动点
- 模块 B：改动点

## 设计与测试目标

- 设计稿：`docs/plans/yyyy-mm-dd-xxx.md`
- 用户确认：已确认 / 未确认，当前为 draft
- 测试目标：真实场景、真实数据、正常路径、异常路径、回归路径和边界路径均已覆盖 / 列出未覆盖原因
```

## 执行提示

当用户要求“提交代码”“推送代码”“发 PR”“整理交付说明”时：

1. 先检查当前分支是否为主干或版本分支。
2. 如果是主干或版本分支，不直接 push，先创建临时分支。
3. 根据仓库历史整理合规的 commit title。
   - 如果用户要的是完整 commit message、合规审查或 shell 命令，加载 `commit-message-zh.md` 与 `shell-commit-examples.md`。
4. 如果是后端新增功能或既有功能变动，先补或更新对应单元测试。
5. 运行与改动相关的单元测试；涉及数据库、消息、协议、跨模块边界、外部依赖、启动装配或事件链路时再跑集成测试。
6. 收集测试通过数、失败数、跳过数、单元测试覆盖率、集成测试结果或不适用原因。
7. 检查是否存在应同步的原始文档；若需要但尚未更新，先补文档或明确记录风险与原因。
8. 检查公共类、SPI / 扩展点方法和复杂逻辑的注释是否符合契约要求，必要时补真实 `@since` / `@see`。
9. 判断本次改动是否过大、是否混入多个主题；如果是，先建议拆分 PR。
10. 如果远端是 GitHub，先检查 `gh auth status`；当前工具支持非沙箱审批时使用非沙箱执行，不支持时说明限制并按可用方式降级。
11. 生成 PR 标题与 PR 描述，确保包含目的、核心变动、测试数据、文档同步情况和风险说明。
12. 创建 PR 前，先确认用户是否认为任务已完成。
13. push 临时分支后，优先使用 `gh pr create` 创建 PR：未完成则 draft，已完成且用户确认后再创建 ready for review；如果 `gh` 不可用，再明确说明阻塞或降级方式。

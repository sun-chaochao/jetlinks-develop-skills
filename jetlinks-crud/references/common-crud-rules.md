# JetLinks 通用 CRUD 规则

本文件面向 JetLinks 系脚手架中的常规 CRUD 需求，重点是复用已有基类与分层，而不是生成大而全的样板代码。

## 适用范围

- 新增或修改 Entity
- 新增或修改 Service
- 新增或修改 Controller
- 标准增删改查
- 基础权限、校验和用户可见文案

## 第一原则：先判断当前模块的编程模型

不要机械询问用户“选响应式还是阻塞式”。

优先级如下：

1. 如果目标模块已经明显统一了风格
   - 跟随现有风格。
   - 通过控制器返回类型、服务基类、仓储接口判断。

2. 如果是在全新模块中且没有任何可参考实现
   - 再询问用户。
   - 如果用户不确定，可建议跟随当前脚手架主流模式。

## 第二原则：最小实现

1. Service 默认只保留确实需要的扩展逻辑
   - 如果基类已经提供标准 CRUD，就不要补一堆无意义方法。

2. Controller 默认只暴露用户要求的接口
   - 如果框架已通过基类或接口暴露通用 CRUD，就不要重复造轮子。

3. 查询能力优先复用已有 `_query` 或 QueryParam 体系
   - 不为每个字段单独造接口。
   - 涉及 `createQuery()`、`createUpdate()`、`createDelete()`、排序、嵌套条件、分页、QueryParam 改写、复杂 SQL 或多查询组合时，读取 `query-dsl-rules.md`。

4. 空模板仓库允许生成首个 CRUD 参考实现
   - 如果目标仓库没有可参考的 Entity / Service / Controller，不要停在“缺少样例”。
   - 允许按本规则生成最小 CRUD 骨架，作为该脚手架中的首个规范样例。

## 推荐工作流

1. 找一个目标模块内最像的 Entity / Service / Controller
2. 复制结构和注解组合
3. 只替换业务字段、路径、权限 ID、文案
4. 如果用户确实要求自定义逻辑，再补服务或控制器方法

如果当前仓库缺少参考实现，改用以下回退顺序：

1. 根 `pom.xml` 与依赖决定技术栈边界
2. 现有少量注解、导入和目录结构决定包路径风格
3. 本规则决定最小 CRUD 骨架
4. 只有关键决策无法判断时才询问用户

## Entity 规则

优先确认：

- 使用哪种基类或接口承载 ID、扩展字段、审计字段
- 是否普遍启用实体事件
- 字段是否需要 JSON、枚举、默认值、只读、校验分组
- 表名和字段长度是否跟随相邻实体

常见做法：

- 通过基类或接口承载通用字段
- 用 `@Schema` 或等价注解描述对外字段
- 用专门注解处理 JSON、枚举、默认值
- 仅在当前模块已有此惯例时补齐创建人、修改人等字段

当没有相邻实体可参考时：

- 生成最小实体，不额外塞入与需求无关字段
- 优先使用最通用的 ID、名称、描述、创建修改时间等模式，但仅限用户需要
- 需要审计字段时，优先选择当前依赖体系已经支持的接口或基类

## Service 规则

优先复用：

- 通用 CRUD 基类
- 树形实体服务基类
- 资产实体服务基类

只在以下情况增加逻辑：

- 保存前校验
- 保存后同步
- 批量更新或批量删除
- 聚合查询
- 调用其他边界能力

## Controller 规则

优先复用：

- 通用 CRUD 控制器接口或基类
- 资产控制器基类
- 树形控制器基类

补充自定义接口的前提：

- 用户明确要求
- 通用 `_query` 无法表达
- 该模块已有同类定制接口模式

当没有相邻控制器可参考时：

- 先生成最小 CRUD 控制器骨架
- 只保留资源路径、权限标识和服务绑定
- 自定义查询、批量操作、联动接口默认不生成

## 权限与文案

1. Controller 上的资源权限 ID 复用模块现有命名方式
2. 自定义动作只有在现有模块使用 `@ResourceAction` 或同类注解时才新增
3. 涉及 CRUD 查询、详情、更新、删除、批量操作、导出或自定义接口时，必须分析是否需要 AssetsHolder 数据权限控制
   - 具体实现切换到 [`../../jetlinks-assets-permission/SKILL.md`](../../jetlinks-assets-permission/SKILL.md)
   - 优先复用 `@AssetsController`、`AssetsHolderCrudController`、`CorrelatesAssetsHolderCrudController`、`AssetsHolder.injectQueryParam`、`AssetsHolder.assertPermission`、`CrudAssetPermission`
   - 不要手写租户、机构、部门、创建人过滤来替代 AssetsHolder
   - 如果资产类型、关联资产字段、权限动作、绑定关系、管理员例外、跨租户或平台上下文拿不准，先询问用户
4. i18n 是否补齐，以当前模块约定为准
   - 若模块已有 i18n 资源并对权限、动作、字段做本地化，则补齐
   - 若模块尚未建立这套约定，不要臆造单独规则
5. 保存前校验、唯一性冲突、资源不存在等用户可见异常，优先沿用当前模块的 `i18nCode` / message key 异常写法
   - 不要直接 `throw new XxxException("名称已存在")`
   - 若本地异常模型只支持 `message`，再在边界层回退到本地化后的 message

## 注释落地

- 新增或修改对外 Entity、DTO、Controller、Service、Command、Event 时，按公共契约补类注释；标准继承型 CRUD 若完全无自定义语义，可沿用相邻模块简化。
- 自定义接口、复杂校验、AssetsHolder 权限边界、兼容旧查询 / 参数、批量上限、复杂 QueryHelper / SQL / DSL 组合，需要在代码旁边写 1 到 3 行短注释说明业务原因或边界。
- 不给简单字段、getter / setter、直接 DTO 搬运、标准基类委托写噪声注释。
- 如果本次 CRUD 改动没有新增任何注释，交付说明必须明确它只涉及标准继承、简单字段或直观委托，没有复杂业务和公共契约变化。

### 低上下文模板仓库下的权限默认规则

如果当前仓库没有可参考的权限命名：

- 资源权限 ID 使用稳定的 `kebab-case`
- 优先由“领域名 + 资源名”组成
- Controller 路径、权限 ID、资源名称保持一一对应
- 只有存在真正的自定义动作时才新增 action 级权限
- 不要在首个 CRUD 骨架中一次性定义过多细粒度权限

## 何时跳到高级规则

遇到以下情况时，继续读取 `advanced-crud-rules.md`：

- 动态复杂查询
- 关联查询
- 批量处理
- 复杂 SQL、原生 SQL 或 QueryHelper
- 性能敏感分页
- CRUD 后强副作用联动

遇到以下情况时，继续读取 `query-dsl-rules.md`：

- `createQuery()` / `QueryParamEntity` DSL 查询
- `createUpdate()` / `createDelete()` 批量更新或删除
- 默认排序、多字段排序、分页稳定排序
- `nest` / `orNest` / `toNestQuery` 嵌套条件
- `setParam` 合并前端筛选与服务端固定条件
- AssetsHolder 查询注入前后的 DSL 组织
- `QueryHelper.transformPageResult` / `combineOneToMany` 多查询结果组合

## 自检清单

- 是否沿用了目标模块的编程模型
- 是否复用了现有 CRUD 基类或接口
- 是否只生成了用户需要的代码
- 是否避免了重复查询接口和重复样板
- 权限、路径、文案是否与当前模块一致
- 是否分析了 AssetsHolder 数据权限：资产类型、权限动作、查询注入、详情访问、更新删除校验、批量操作和关联资产边界
- 是否为公共类、自定义接口、复杂校验、权限边界、兼容逻辑、批量上限或复杂查询补了必要代码注释
- 若处于空模板仓库，是否明确区分了“来自现有工作区的事实”和“来自通用规则的默认决策”

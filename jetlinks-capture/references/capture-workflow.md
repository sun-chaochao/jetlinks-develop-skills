# JetLinks 知识沉淀工作流

本文件用于在任务结束后判断：本次工作是否值得沉淀，以及应当沉淀成什么形式。

## 核心原则

1. 不是每次改动都要写文档
   - 只有当本次工作产生了稳定、可复用、非显然的知识时，才值得沉淀。

2. 先选最小沉淀形式
   - 单次任务总结优先写 `worklog`
   - 项目经验优先写 `knowledge`
   - 稳定执行流程优先写 `playbook`
   - 长期稳定且跨任务通用的规则，再考虑更新 prompt 或 skill
   - 如果已有归属文档可以承载，优先更新已有文档，不另起碎片。

3. 沉淀的是“规则”和“经验”，不是简单 diff
   - 要写为什么这样做、以后遇到什么场景应复用，而不是仅列出改了哪些文件。

4. 先建议，再生成
   - 任务完成后，如果判断值得沉淀，智能体应先给出“是否建议沉淀、推荐形式、推荐路径、摘要草稿”。
   - 用户确认后再生成正式文档；用户已明确要求直接生成时可直接落文档。
   - README 只放仓库或模块长期总览，不放单次任务总结、测试报告、排查流水或 PR 摘要。

5. 可抽成 skill 时，再给一次升级选择
   - 如果结论已经不再局限于当前项目，而是对多个 JetLinks 项目都成立，可以继续询问是否并入 skill。
   - 如果用户同意，应将目标指向 `https://github.com/jetlinks/jetlinks-develop-skills`，并按该仓库结构准备 PR。

## 何时值得沉淀

满足以下任意两项，建议沉淀：

- 踩坑明显，后续很容易重复踩坑
- 结论依赖项目上下文，不属于通用常识
- 同类模式已经出现两次以上
- 未来智能体大概率会重复遇到
- 结论会影响模块边界、事件模型、响应式链路、测试方式或交付流程
- 当前结论相对稳定，不是短期过渡方案

不建议沉淀：

- 一次性业务改动
- 很快会废弃的方案
- 只有文件改动，没有抽象出可复用规则
- 仍未验证、结论尚不稳定
- 已经可以在 PR 描述、CI 报告或已有归属文档中表达清楚的临时信息

## 输出形式

### 1. worklog

适用：

- 单次任务总结
- 便于追溯“这次做了什么、为什么这样做、验证了什么”

建议内容：

- 背景
- 核心变动
- 验证结果
- 是否值得继续沉淀为 knowledge 或 playbook

### 2. knowledge

适用：

- 项目特有约定
- 排坑经验
- 架构决策、模块边界、命名规则、依赖约束

建议内容：

- 适用场景
- 结论
- 依据
- 风险与例外

### 3. playbook

适用：

- 稳定且可重复执行的操作流程
- 发布、测试、调试、迁移、模块创建、知识沉淀流程

建议内容：

- 触发条件
- 步骤
- 输入与输出
- 常见失败点

### 4. prompt / skill update

适用：

- 已经反复出现
- 明显会影响后续大量任务
- 足够稳定，值得变成智能体默认流程的一部分

### 5. official skill PR

适用：

- 已经可以抽象成跨项目复用的 JetLinks skill
- 不依赖某个单仓库的私有目录、模块名或一次性业务背景
- 更适合进入 `jetlinks-develop-skills`，而不是只留在当前项目 `.ai/` 或本地 prompt 中

建议动作：

- 先给用户一个明确选择：是否并入 skill
- 若用户同意，再整理为新增 skill 或更新现有 skill
- 准备 PR 到 `https://github.com/jetlinks/jetlinks-develop-skills`

## 标准收尾流程

1. 完成任务本身
2. 判断是否产出了稳定、可复用、非显然的知识
3. 若值得沉淀，主动提示用户是否需要整理为文档
4. 给出推荐形式、目标路径和摘要草稿
5. 用户确认后，优先更新已有归属文档；否则再写入 `.ai/` 的统一目录
6. 如果结论已跨任务稳定，再考虑回写 prompt 或 skill
7. 如果结论已跨项目稳定，再询问是否并入 `jetlinks-develop-skills` 并提交官方 PR

## 推荐目录

统一目录建议如下：

```text
.ai/
  worklog/
  knowledge/
  playbooks/
  templates/
```

推荐文件名：

- `worklog/YYYY-MM-DD-topic.md`
- `knowledge/<domain>-<topic>.md`
- `playbooks/<workflow>.md`

不推荐：

- `README.md`：不放单次经验、测试报告或排查记录。
- `docs/*-test-report.md`：除非项目已有测试报告目录或用户明确要求。
- `summary.md`、`todo.md`、`note.md`：命名不可检索，容易形成碎片。

## 标准收尾动作

任务结束时优先输出：

1. 是否建议沉淀：建议 / 不建议
2. 理由
3. 推荐沉淀形式
4. 推荐路径
5. 摘要草稿
6. 如果用户确认，再生成正式文档
7. 如果已具备 skill 抽象价值，询问是否并入 `jetlinks-develop-skills` 并提交 PR 到官方仓库

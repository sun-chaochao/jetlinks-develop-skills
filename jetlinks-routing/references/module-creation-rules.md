# JetLinks 模块创建规则

本文件用于在 JetLinks 系脚手架中创建新模块，但不会假设固定目录、固定 groupId、固定版本号或固定模块命名。

## 核心原则

1. 先继承当前脚手架结构，不自创结构
   - 先看根 `pom.xml`、相邻聚合模块、同类业务模块。

2. 只有在用户明确要求时才创建额外内容
   - 默认只创建模块结构和必要骨架。
   - 不自动附带示例实体、演示控制器、测试样例。

3. 新模块形态跟随现有脚手架
   - 如果现有脚手架偏向单模块，就继续单模块。
   - 如果现有脚手架常见聚合 + core + manager，就沿用该结构，并保持职责语义：`manager` 承载管理端 CRUD 与运行时装配，`core` 只承载可复用领域模型、DTO、命令 / 事件契约、常量、扩展接口等公共定义。

4. 空模板仓库允许生成首个参考模块
   - 如果仓库里没有任何可直接参考的业务模块，不要停止。
   - 允许基于当前聚合结构和本规则生成“最小首个模块”，让它成为后续模块的参考样板。

## 先确认四件事

1. 新模块应该挂到哪个根目录
2. 当前脚手架常见的模块形态
3. 父 `pom`、版本继承和模块注册方式
4. 包名、资源路径、自动配置路径的既有惯例

## 常见模块形态

### 单模块

适用：
- 纯本地业务
- 简单 CRUD
- 没有明确的对外契约层

说明：
- 如果当前脚手架已经明确拆分为 `manager` / `core`，不要把“简单 CRUD”落到 `core`；CRUD 仍应放入 `manager`。

### 聚合模块

适用：
- 当前脚手架普遍使用父模块聚合多个子模块
- 一个能力同时包含 manager、core、adapter、plugin 等子模块

常见职责：
- `*-manager`：承载管理端 CRUD、Controller / Web 接口、应用 Service、持久化 Entity / Repository / DAO、权限校验、i18n、运行时配置和自动装配。
- `*-core`：承载跨模块可复用的领域对象 / 值对象、DTO、命令定义、事件定义、常量、扩展接口和 SPI 契约。
- 不因“有 CRUD”创建或污染 `core`；只有存在被其他模块复用的公共定义时才创建或修改 `core`。

### 契约 + 实现分离

适用：
- 需要对外暴露可被其他模块稳定依赖的 DTO、命令、事件、接口
- 需要减少其他模块对实现细节的依赖

约束：
- 契约层通常应落在既有 `core` 模块，而不是额外创建 `api` 模块。
- 不因“以后可能复用”“存在 DTO”“存在接口”拆出 `api` 或把 CRUD 实现放入 `core`。
- 只有用户明确要求、当前同类模块已有 `api + manager/core` 形态且必须跟随、或该契约需要独立发布 / 跨服务消费时，才考虑新增 `api`。

## 低上下文模板仓库模式

当没有相邻模块可参考时，按以下顺序决策：

1. 先继承根 `pom.xml` 和最近的聚合方式
2. 如果无法判断是否需要拆分，默认选择最小可用结构
3. 只有当用户明确要求跨边界契约层时，再拆分为多子模块

默认建议：

- 先创建一个最小业务实现模块；若脚手架存在 `manager` 语义，业务实现模块优先命名为 `*-manager`
- 带上最基本的 `pom.xml`、源码目录、资源目录
- 如脚手架已有自动配置导出机制，再补自动配置文件
- 不自动创建演示实体和测试数据
- 不默认创建 `*-api`；不默认创建 `*-core`。只有明确需要公共领域 / DTO / 命令 / 事件契约时，才补 `*-core`

## 创建流程

1. 选模板来源
   - 找一个最接近的新模块参考对象。

2. 复制结构，不复制业务
   - 继承目录、`pom.xml` 形态、资源目录、自动配置文件布局。

3. 补模块注册
   - 如果父 `pom.xml` 通过 `<modules>` 聚合，记得注册新模块。
   - 如果有自动配置导出文件，也按现有方式补充。

4. 创建最小骨架
   - 只创建用户明确要求的 `configuration`、`entity`、`service`、`web`、`command`、`i18n` 等目录和文件。
   - CRUD 相关的 `entity`、`service`、`web`、`repository`、权限和 i18n 目录应放在 `manager` 或当前实现模块，不放在 `core`。
   - `core` 目录只创建公共 `domain`、`dto`、`command`、`event`、`constant`、`spi` / `api` 契约等目录；没有公共契约时不要创建空 `core`。
   - 新建公共配置类、SPI 入口、契约接口或首个参考类时，补类注释说明模块职责、边界和调用方；不要生成无意义示例注释。

5. 当用户只说“创建一个模块”
   - 默认只完成模块结构、注册和必要配置。
   - 不主动附带 CRUD，除非用户明确提出“创建基础 CRUD”；如果需要 CRUD，优先落到 `manager` 或当前实现模块。

## `pom.xml` 规则

1. 不硬编码版本号
   - 继承最近的父 `pom` 或使用当前工作区已有属性。

2. 不臆造依赖
   - 先从相邻模块复制基础依赖，再按实际功能加最少依赖。

3. 依赖方向要一致
   - `core` / 契约模块不要反向依赖 `manager` / 实现模块。
   - `manager` 可以依赖 `core` 中的公共 DTO、命令、事件和领域契约；`core` 不依赖 CRUD、Web、Repository、运行时管理实现。

## 资源与配置

按当前脚手架判断是否需要：

- `META-INF/spring/...AutoConfiguration.imports`
- `i18n/.../messages_zh.properties`
- `i18n/.../messages_en.properties`
- `application.yml` 片段
- SPI/插件注册文件

## 输出模板

分析时输出：

1. 推荐的新模块形态
2. 参考了哪个现有模块
3. 需要创建和注册的文件
4. 哪些内容不会自动生成
5. 新建公共契约或首个参考类的注释落点
